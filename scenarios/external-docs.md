# Scenario: External-Facing Documentation

Use this for release notes, customer explanations, design summaries, support
responses, and externally consumable feature docs.

## Flow

1. Identify audience and purpose.
2. Separate confirmed behavior from planned or experimental behavior.
3. Verify technical claims against code, tests, accepted design, or build
   output.
4. Write concise user-facing content.
5. Review for over-promising, unsupported claims, and missing limitations.
6. Link internal evidence when the document is not public.

## Output Types

- release note
- issue explanation
- feature usage note
- risk and limitation summary
- upgrade or compatibility note

## Rules

- Do not expose internal uncertainty as external commitments.
- Do not claim performance gains without measurement context.
- Describe limitations clearly when behavior is gated or experimental.
